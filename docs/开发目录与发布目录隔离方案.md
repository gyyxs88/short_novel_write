# 开发目录与发布目录隔离方案

这份文档解决一个很实际的问题：

- 开发目录里允许保留本地敏感配置
- 推送到 GitHub 的发布目录必须是脱敏后的

结论先说：

- 不要把发布仓库做成开发仓库的子目录
- 推荐使用“兄弟目录”结构
- 推荐把“发布目录”作为唯一对外 GitHub 仓库
- 开发目录只保留本地 Git 提交，不承担对外推送职责
- 用同步脚本把开发目录内容复制到发布目录
- 本地敏感配置统一放到 `local/`，不要进 Git

## 为什么不能用 `A/B` 嵌套仓库

如果开发目录是 `A`，发布目录放成 `A/B`，看起来省事，实际风险很高：

- `A` 和 `A/B` 都是 Git 仓库，长期维护时很容易混淆
- 一旦 `A` 里没有正确忽略 `B/`，容易出现误跟踪、子模块状态混乱、误提交
- 同步脚本如果路径判断没做好，容易把开发目录和发布目录互相覆盖
- 最重要的是，敏感文件只要进过开发仓库的 Git 历史，后面再删也不算安全

所以这个仓库的约定是：

- 发布目录必须放在开发目录外部
- 最稳妥的是平级兄弟目录

## 推荐目录结构

```text
D:/Project/
├─ short_novel_write/          # 开发目录
└─ short_novel_write_publish/  # 发布目录 / GitHub 仓库
```

推荐做法：

1. `short_novel_write/` 继续作为你的真实开发目录
2. `short_novel_write_publish/` 作为唯一对外 GitHub 仓库
3. `short_novel_write/` 只做本地 Git 提交，不直接推远程
4. 每完成一个阶段，用同步工具把脱敏后的内容同步过去
5. 推送前只检查 `short_novel_write_publish/`

## 推荐的长期工作流

如果你的目标是“开发目录保留完整本地上下文，GitHub 上只保留脱敏后的发布版本”，推荐长期固定为下面这套：

1. 开发目录负责真实开发
2. 开发目录可以继续本地 `git commit`
3. 开发目录不要承担 `git push` 责任
4. 发布目录作为唯一对外推送目录
5. 每次发布前，先从开发目录同步到发布目录
6. 只在发布目录做对外 `commit / push`

这个模式的优点是：

- 敏感配置和本地过程文件天然留在开发目录
- GitHub 上只保留脱敏后的可公开内容
- 以后你只需要把“能不能推”这个判断放在发布目录，不用每次担心开发目录误推

## Git 远程约定

如果采用上面的长期工作流，建议把 Git 远程职责也固定下来：

- 开发目录：允许本地提交，但不保留 GitHub `origin`
- 发布目录：保留 GitHub `origin`，负责 `main` 和其他对外分支

也就是说：

- 开发目录是“本地研发仓库”
- 发布目录是“对外发布仓库”

这样最符合你的目标，也最能减少误操作风险。

## 分支约定

这部分必须说死，不要留模糊空间：

- 开发目录允许临时建分支，但默认只留本地，不对外推送。
- 任何临时分支上的有效改动，必须先合并、`cherry-pick` 或用其他明确方式收回开发目录 `main`。
- 只有开发目录 `main` 上已经确认要发布的内容，才允许同步到发布目录。
- 发布目录只保留对外主线，默认就是 `main`；不要把验收分支、备份分支、临时排查分支推到发布仓库。
- 如果只是想留一个历史点，优先用本地分支、tag 或 GitHub Release，不要在发布仓库里长期挂备份分支。

一句话约束：

- 临时分支可以有，但不能直接发布。
- 对外发布只能从 `main` 走。

### 为什么要这么约束

- 临时分支一旦推到发布仓库，GitHub 页面就会暴露额外分支，和“发布仓库只代表稳定对外版本”这个目标冲突。
- 备份分支和验收分支如果长期留在发布仓库，很快就会分不清哪些是正式版本、哪些只是中间态。
- 从临时分支直接同步发布目录，会把“还没并回主线的内容”伪装成正式发布结果，后面很难追账。

### 允许的发布前状态

开始同步前，开发目录应满足：

- 当前分支是 `main`
- 要发布的改动已经在开发目录 `main`
- 开发目录没有需要额外解释的临时分支现场

发布目录应满足：

- 当前分支是 `main`
- 只承担同步后的 `commit / push`
- 不承接开发目录的验收分支、备份分支和临时分支

## 本地私有目录约定

仓库根目录新增了 `.gitignore` 规则：

- `local/`

用途：

- 保存不会推 Git 的本地配置
- 保存本地发布同步配置
- 保存本地环境变量加载脚本

例如：

```text
short_novel_write/
├─ local/
│  ├─ publish.ps1
│  ├─ publish_sync.local.json
│  └─ dev_env.ps1
```

`dev_env.ps1` 可以像这样：

```powershell
$env:DEEPSEEK_API_KEY = "你的 key"
$env:OPENROUTER_API_KEY = "你的 key"
```

## 发布同步工具

新增工具：

- `tools/publish_repo_sync.py`

这个工具会做几件事：

- 从当前开发仓库同步文件到外部发布目录
- 默认排除 `AGENTS.md`、`docs/memory/`、`docs/superpowers/`、`zhihu-yanxuan-short-story2/`、`tools/dev_todoist_cli.py`、`tests/tools/test_dev_todoist_cli.py`、`outputs/`、`temp/`、`local/`、`.venv/`、`.git/`、`.env*` 等内容
- 支持额外自定义排除规则
- 会直接拦截“发布目录在开发目录里面”这种危险用法
- 会报告发布目录里存在但开发目录里已经没有的陈旧文件
- 会额外报告“当前已被排除、但发布目录里还残留着的旧文件”

## 推荐配置文件

本地创建：

- `local/publish_sync.local.json`

示例内容：

```json
{
  "target_dir": "../short_novel_write_publish"
}
```

说明：

- `target_dir` 建议写成开发目录根目录的兄弟目录
- `exclude_patterns` 用来补充默认排除项
- 如果不传 `exclude_patterns`，工具会直接使用默认排除名单

## 使用方式

### 1. 先预演，不实际复制

```powershell
.\.venv\Scripts\python.exe tools\publish_repo_sync.py --config local\publish_sync.local.json --dry-run
```

### 2. 确认无误后正式同步

```powershell
.\.venv\Scripts\python.exe tools\publish_repo_sync.py --config local\publish_sync.local.json
```

### 3. 如果不想写配置文件，也可以直接传目标目录

```powershell
.\.venv\Scripts\python.exe tools\publish_repo_sync.py --target-dir ..\short_novel_write_publish --dry-run
```

## 固定发布命令

如果你以后就按“开发目录本地提交，发布目录对外推送”这套模式走，最常用的命令就是下面这一组。

### 1. 在开发目录同步到发布目录

```powershell
Set-Location D:\Project\short_novel_write
.\.venv\Scripts\python.exe tools\publish_repo_sync.py --config local\publish_sync.local.json
```

### 2. 在发布目录检查并提交

```powershell
Set-Location D:\Project\short_novel_write_publish
git status
git add .
git commit -m "chore: update publish snapshot"
git push
```

### 3. 每次发布时的最短操作顺序

```powershell
Set-Location D:\Project\short_novel_write
.\.venv\Scripts\python.exe tools\publish_repo_sync.py --config local\publish_sync.local.json
Set-Location D:\Project\short_novel_write_publish
git status
git add .
git commit -m "chore: update publish snapshot"
git push
```

### 4. 一键发布脚本

如果你不想每次手动敲完整套命令，可以直接使用本地脚本：

```powershell
Set-Location D:\Project\short_novel_write
.\local\publish.ps1
```

常用变体：

```powershell
.\local\publish.ps1 -DryRun
.\local\publish.ps1 -CommitMessage "feat: publish latest llm pipeline updates"
.\local\publish.ps1 -SkipPush
```

补充说明：

- 第一段命令必须在开发目录执行
- 第二段命令必须在发布目录执行
- 如果 `git status` 为空，说明这次没有需要发布的变更
- 提交信息可以按实际改动改写，不必固定使用示例内容
- `publish.ps1` 默认要求发布目录当前分支必须是 `main`

## GitHub Releases 发布包

如果你的目标是“把干净源码包分发给别人”，推荐做法是：

1. 发布目录继续只放源码快照
2. 不要在发布仓库里新增 `releases/` 目录保存 zip
3. 发版时使用 GitHub Releases
4. 发布包作为 Release 附件上传，不进入仓库正文

这样做的原因很简单：

- 发布仓库负责源码历史
- 发布包负责分发
- 二进制 zip 不会把 Git 历史越堆越大
- 下次同步发布目录时，也不会把你手工放进去的 zip 误报成陈旧文件

### 1. 只需要源码快照时

如果你只需要“当前发布仓库的干净源码 zip”，其实可以直接用 GitHub Releases 自动生成的：

- `Source code (zip)`
- `Source code (tar.gz)`

前提是你已经在发布仓库打好 tag 并创建 release。

### 2. 需要自定义干净 zip 时

如果你想自己生成一个更可控的 zip 再上传到 GitHub Releases，可以使用：

- `tools/release_package_builder.py`

示例：

```powershell
Set-Location D:\Project\short_novel_write
.\.venv\Scripts\python.exe tools\release_package_builder.py `
  --source-dir ..\short_novel_write_publish `
  --output-dir ..\release_artifacts `
  --release-name short_novel_write-v0.1.0
```

约定：

- `source-dir` 指向发布目录
- `output-dir` 必须放在发布目录外部
- 不要把输出目录放进发布仓库，否则很容易又回到“仓库里塞 zip 包”的老路

这个工具默认会排除：

- `.git/`
- `__pycache__/`
- `.pytest_cache/`
- `.venv/`
- `local/`
- `outputs/`
- `temp/`
- `.env*`
- `*.pyc`

### 3. 推荐发布顺序

```powershell
Set-Location D:\Project\short_novel_write
.\local\publish.ps1 -CommitMessage "chore: update publish snapshot"
.\.venv\Scripts\python.exe tools\release_package_builder.py `
  --source-dir ..\short_novel_write_publish `
  --output-dir ..\release_artifacts `
  --release-name short_novel_write-v0.1.0
Set-Location D:\Project\short_novel_write_publish
git tag v0.1.0
git push origin main --tags
```

然后去 GitHub：

1. 基于 `v0.1.0` 创建 Release
2. 如果只要源码快照，直接用 GitHub 自动生成的 Source code
3. 如果要自定义 zip，把 `..\release_artifacts\short_novel_write-v0.1.0.zip` 作为附件上传

## 同步前后的检查建议

同步前：

- 到开发目录执行 `git branch --show-current`，确认当前分支是 `main`
- 确认发布目录不是开发目录的子目录
- 确认敏感文件放在 `local/`、`outputs/` 或其他已忽略位置
- 先跑一次 `--dry-run`

同步后：

- 到发布目录执行 `git branch --show-current`，确认当前分支还是 `main`
- 到发布目录里执行 `git status`
- 抽查 `README.md`、`tools/`、`tests/`、`docs/`
- 确认没有 `.env`、本地 key、SQLite、输出文件、私有记忆文档

## 当前约定的边界

这个工具当前只做“安全同步”和“陈旧文件提示”：

- 会复制需要发布的文件
- 会报告发布目录里的陈旧文件
- 会报告已经进入排除名单、但仍残留在发布目录里的旧文件
- 不会自动删除陈旧文件

这样做是为了降低误删风险。

如果后面确定发布目录已经稳定，也可以再继续补“安全清理陈旧文件”的能力。
