# 开发目录与发布目录隔离方案

这份文档解决一个很实际的问题：

- 开发目录里允许保留本地敏感配置
- 推送到 GitHub 的发布目录必须是脱敏后的

结论先说：

- 不要把发布仓库做成开发仓库的子目录
- 推荐使用“兄弟目录”结构
- 推荐把“发布目录”作为唯一对外 GitHub 仓库
- 开发目录只保留本地 Git 提交，不承担对外推送职责
- 用同步脚本把开发目录内容复制到发布目录
- 本地敏感配置统一放到 `local/`，不要进 Git

## 为什么不能用 `A/B` 嵌套仓库

如果开发目录是 `A`，发布目录放成 `A/B`，看起来省事，实际风险很高：

- `A` 和 `A/B` 都是 Git 仓库，长期维护时很容易混淆
- 一旦 `A` 里没有正确忽略 `B/`，容易出现误跟踪、子模块状态混乱、误提交
- 同步脚本如果路径判断没做好，容易把开发目录和发布目录互相覆盖
- 最重要的是，敏感文件只要进过开发仓库的 Git 历史，后面再删也不算安全

所以这个仓库的约定是：

- 发布目录必须放在开发目录外部
- 最稳妥的是平级兄弟目录

## 推荐目录结构

```text
D:/Project/
├─ short_novel_write/          # 开发目录
└─ short_novel_write_publish/  # 发布目录 / GitHub 仓库
```

推荐做法：

1. `short_novel_write/` 继续作为你的真实开发目录
2. `short_novel_write_publish/` 作为唯一对外 GitHub 仓库
3. `short_novel_write/` 只做本地 Git 提交，不直接推远程
4. 每完成一个阶段，用同步工具把脱敏后的内容同步过去
5. 推送前只检查 `short_novel_write_publish/`

## 推荐的长期工作流

如果你的目标是“开发目录保留完整本地上下文，GitHub 上只保留脱敏后的发布版本”，推荐长期固定为下面这套：

1. 开发目录负责真实开发
2. 开发目录可以继续本地 `git commit`
3. 开发目录不要承担 `git push` 责任
4. 发布目录作为唯一对外推送目录
5. 每次发布前，先从开发目录同步到发布目录
6. 只在发布目录做对外 `commit / push`

这个模式的优点是：

- 敏感配置和本地过程文件天然留在开发目录
- GitHub 上只保留脱敏后的可公开内容
- 以后你只需要把“能不能推”这个判断放在发布目录，不用每次担心开发目录误推

## Git 远程约定

如果采用上面的长期工作流，建议把 Git 远程职责也固定下来：

- 开发目录：允许本地提交，但不保留 GitHub `origin`
- 发布目录：保留 GitHub `origin`，负责 `main` 和其他对外分支

也就是说：

- 开发目录是“本地研发仓库”
- 发布目录是“对外发布仓库”

这样最符合你的目标，也最能减少误操作风险。

## 本地私有目录约定

仓库根目录新增了 `.gitignore` 规则：

- `local/`

用途：

- 保存不会推 Git 的本地配置
- 保存本地发布同步配置
- 保存本地环境变量加载脚本

例如：

```text
short_novel_write/
├─ local/
│  ├─ publish_sync.local.json
│  └─ dev_env.ps1
```

`dev_env.ps1` 可以像这样：

```powershell
$env:DEEPSEEK_API_KEY = "你的 key"
$env:OPENROUTER_API_KEY = "你的 key"
```

## 发布同步工具

新增工具：

- `tools/publish_repo_sync.py`

这个工具会做几件事：

- 从当前开发仓库同步文件到外部发布目录
- 默认排除 `AGENTS.md`、`docs/memory/`、`docs/superpowers/`、`zhihu-yanxuan-short-story2/`、`tools/dev_todoist_cli.py`、`tests/tools/test_dev_todoist_cli.py`、`outputs/`、`temp/`、`local/`、`.venv/`、`.git/`、`.env*` 等内容
- 支持额外自定义排除规则
- 会直接拦截“发布目录在开发目录里面”这种危险用法
- 会报告发布目录里存在但开发目录里已经没有的陈旧文件
- 会额外报告“当前已被排除、但发布目录里还残留着的旧文件”

## 推荐配置文件

本地创建：

- `local/publish_sync.local.json`

示例内容：

```json
{
  "target_dir": "../short_novel_write_publish"
}
```

说明：

- `target_dir` 建议写成开发目录根目录的兄弟目录
- `exclude_patterns` 用来补充默认排除项
- 如果不传 `exclude_patterns`，工具会直接使用默认排除名单

## 使用方式

### 1. 先预演，不实际复制

```powershell
.\.venv\Scripts\python.exe tools\publish_repo_sync.py --config local\publish_sync.local.json --dry-run
```

### 2. 确认无误后正式同步

```powershell
.\.venv\Scripts\python.exe tools\publish_repo_sync.py --config local\publish_sync.local.json
```

### 3. 如果不想写配置文件，也可以直接传目标目录

```powershell
.\.venv\Scripts\python.exe tools\publish_repo_sync.py --target-dir ..\short_novel_write_publish --dry-run
```

## 同步前后的检查建议

同步前：

- 确认发布目录不是开发目录的子目录
- 确认敏感文件放在 `local/`、`outputs/` 或其他已忽略位置
- 先跑一次 `--dry-run`

同步后：

- 到发布目录里执行 `git status`
- 抽查 `README.md`、`tools/`、`tests/`、`docs/`
- 确认没有 `.env`、本地 key、SQLite、输出文件、私有记忆文档

## 当前约定的边界

这个工具当前只做“安全同步”和“陈旧文件提示”：

- 会复制需要发布的文件
- 会报告发布目录里的陈旧文件
- 会报告已经进入排除名单、但仍残留在发布目录里的旧文件
- 不会自动删除陈旧文件

这样做是为了降低误删风险。

如果后面确定发布目录已经稳定，也可以再继续补“安全清理陈旧文件”的能力。
