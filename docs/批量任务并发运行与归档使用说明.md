# 批量任务并发运行与归档使用说明

## 作用

`tools/story_batch_runner.py` 用来批量跑“多小说同时生成”这件事。

它默认做的是这条链：

1. 读取 jobs JSON
2. 为每个 job 建独立运行目录
3. 为每个 job 初始化独立 `story_ideas.sqlite3`
4. 并发执行单 job 写作链
5. 为每个 job 输出单独的 `report.json / report.md`
6. 用单线程归档 worker 串行写入 `archive.sqlite3`
7. 归档成功后按配置删除源运行库

这套工具的重点不是替代单 job CLI，而是把“并发生成 + 归档收口”收成一个稳定入口。

## 相关文件

- 批量执行入口：`tools/story_batch_runner.py`
- 单任务归档器：`tools/story_archive_manager.py`
- 单 job 执行链：`tools/story_regression_runner.py`
- job 文件结构：本工具自己解析 JSON，不依赖内置样本集

## 当前默认约定

### 1. 每个 job 独立运行库

批量 runner 不会让多个 job 共用同一个业务 SQLite。

每个 job 都会落到自己的目录，例如：

- `outputs/batch/<batch_name>/jobs/<job_id>/story_ideas.sqlite3`

### 2. 归档单线程收口

即使前面的生成 worker 并发跑，归档阶段也只有一个线程负责往 `archive.sqlite3` 写。

这样做是为了避免把归档库重新变成并发冲突点。

### 3. jobs 文件支持 BOM

当前 loader 会按 `utf-8-sig` 读取 jobs 文件。

也就是说，如果你用 PowerShell `Set-Content -Encoding UTF8` 生成 jobs JSON，即使带 BOM 也能正常读。

## jobs 文件格式

支持两种顶层格式：

### 直接传数组

```json
[
  {
    "job_id": "job_001",
    "style": "zhihu",
    "prompt": "婚礼前夜，女主收到失踪前任的求救短信。",
    "target_char_range": [8000, 12000],
    "target_chapter_count": 6,
    "candidate_count": 3,
    "plan_count": 3,
    "idea_pack_route": {
      "generation_mode": "llm",
      "llm_environment": "kxaug_default"
    },
    "plan_route": {
      "generation_mode": "llm",
      "llm_environment": "kxaug_default"
    },
    "draft_route": {
      "generation_mode": "llm",
      "llm_environment": "kxaug_default"
    },
    "notes": "知乎强钩子长稿测试"
  }
]
```

### 用对象包一层 `jobs`

```json
{
  "jobs": [
    {
      "job_id": "job_001",
      "style": "zhihu",
      "prompt": "婚礼前夜，女主收到失踪前任的求救短信。"
    }
  ]
}
```

## 支持字段

每个 job 当前支持这些字段：

- `job_id`
- `style`
- `prompt`
- `target_char_range`
- `target_chapter_count`
- `candidate_count`
- `plan_count`
- `selected_plan_variant_index`
- `notes`
- `tags`
- `idea_pack_route`
- `plan_route`
- `draft_route`

其中 `idea_pack_route / plan_route / draft_route` 的结构和回归 runner 一样，支持：

- `generation_mode`
- `llm_environment`
- `provider`
- `model`
- `api_mode`

## 默认路由

如果 job 没显式写 route，当前默认是：

- `idea_pack_route`：`deterministic`
- `plan_route`：
  - `zhihu` -> `llm_environment=zhihu_plan_default`
  - `douban` -> `llm_environment=douban_plan_default`
- `draft_route`：
  - `zhihu` -> `llm_environment=zhihu_draft_default`
  - `douban` -> `llm_environment=douban_draft_default`

所以：

- 只跑 deterministic 批次时，建议把三个 route 都显式写成 `deterministic`
- 跑 LLM 批次时，建议准备好模板库并显式写清环境名，避免误用默认环境

## 参数校验

当前会在读取 jobs 文件时先做一轮前置校验。

比如：

- `job_id` 不能为空，也不能带路径分隔符
- `target_char_range` 必须是两个递增正整数
- `target_chapter_count` 必须符合正文链允许范围
- `plan_count` 会按当前方案生成器规则校验，非法时直接拒绝，不等跑到半路再失败

## 运行方式

### 最常见用法

```powershell
$env:PYTHONDONTWRITEBYTECODE='1'
.\.venv\Scripts\python.exe -B tools\story_batch_runner.py `
  --jobs-file temp\jobs.json `
  --output-root outputs\batch `
  --run-name novel_batch_001 `
  --template-db outputs\templates\template.sqlite3 `
  --archive-db outputs\archive\archive.sqlite3 `
  --max-workers 3
```

### 归档后保留源运行库

默认归档成功后会删除每个 job 的运行库。

如果你想先保留，显式加：

```powershell
$env:PYTHONDONTWRITEBYTECODE='1'
.\.venv\Scripts\python.exe -B tools\story_batch_runner.py `
  --jobs-file temp\jobs.json `
  --keep-source-db
```

### 有 job 失败就返回退出码 1

```powershell
$env:PYTHONDONTWRITEBYTECODE='1'
.\.venv\Scripts\python.exe -B tools\story_batch_runner.py `
  --jobs-file temp\jobs.json `
  --fail-on-job-failure
```

### 有归档失败就返回退出码 1

```powershell
$env:PYTHONDONTWRITEBYTECODE='1'
.\.venv\Scripts\python.exe -B tools\story_batch_runner.py `
  --jobs-file temp\jobs.json `
  --fail-on-archive-failure
```

## 输出目录

批量 runner 会生成两层报告。

### 1. 批次级报告

在批次根目录下：

- `batch_report.json`
- `batch_report.md`

它们用来看：

- 总 job 数
- 通过/失败数量
- 归档成功/失败数量
- 总耗时
- 总 token
- 每个 job 的最终状态和归档状态

### 2. 单 job 报告

每个 job 自己也会有：

- `jobs/<job_id>/report.json`
- `jobs/<job_id>/report.md`

它们用来看这部小说自己的阶段细节。

## 模板库怎么用

如果你要跑 LLM 路线，推荐传：

- `--template-db`

batch runner 会为每个 job 复制一份模板库，作为它自己的运行库起点。

如果不传模板库：

- 会创建 fresh SQLite
- deterministic 路线能直接跑
- 需要 LLM 配置的 job 可能会因缺少环境配置而失败

## 当前限制

### 1. 归档仍然依赖单 job report 结构

批量 runner 写出的每个 job 报告，底层还是单 case report 格式，这样归档器才能直接复用。

### 2. 当前并发是 job 级并发

也就是说：

- 不会按阶段拆不同并发池
- 一个 job 仍然是完整串行跑完自己的链路

这已经足够支持“多部小说同时生成”，但还不是更细粒度的阶段限流调度。

## 适合的使用场景

- 同时批量生成多部小说
- 每个 job 都要求独立运行库
- 跑完后希望统一归档，避免 SQLite 文件无限增长
- 需要批次级和单 job 级的双层报告
