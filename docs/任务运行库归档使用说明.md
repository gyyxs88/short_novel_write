# 任务运行库归档使用说明

## 作用

`tools/story_archive_manager.py` 用来把单次小说任务目录中的运行库和报告，归档到统一的 `archive.sqlite3`。

它解决的是两个问题：

- 运行阶段需要“每个任务独立 SQLite”，避免并发串库
- 任务结束后又需要“统一可检索的沉淀库”，避免运行库文件无限膨胀

归档器当前的输入是一个单任务目录，目录里至少要有：

- `story_ideas.sqlite3`
- `report.json`

## 推荐分层

当前推荐按三层库来用：

### 1. 模板库

只放稳定配置，不跑业务数据。

典型内容：

- LLM provider
- LLM model
- LLM environment

建议命名：

- `template.sqlite3`

### 2. 运行库

每个小说任务一份，只服务当前 job。

典型内容：

- `idea_card_batches`
- `idea_cards`
- `idea_batch_cards`
- `idea_packs`
- `idea_pack_evaluations`
- `story_plans`
- `story_payloads`
- `story_drafts`

建议路径形态：

- `outputs/runs/<job_id>/story_ideas.sqlite3`

### 3. 归档库

统一长期沉淀库，不参与创作过程。

当前归档表包括：

- `archive_jobs`
- `archive_stage_runs`
- `archive_idea_card_batches`
- `archive_idea_cards`
- `archive_idea_batch_cards`
- `archive_idea_packs`
- `archive_idea_pack_evaluations`
- `archive_story_plans`
- `archive_story_payloads`
- `archive_story_drafts`

## 当前归档内容

归档器不是只存最后成品，而是会把整条候选链一起写进去。

当前会归档：

- 选题卡与卡组批次
- 创意包与评测结果
- 故事方案
- story payload
- 正文草稿
- 最终选中的 pack / plan / payload / draft 标记
- `report.json`
- 各阶段耗时
- 各阶段 token 统计
- inspect 结果

其中最终成品正文会直接写入：

- `archive_story_drafts.content_markdown`

所以后续检索、统计、精选筛稿不需要回头读运行库。

## 运行方式

### 直接调用归档脚本

```powershell
$env:PYTHONDONTWRITEBYTECODE='1'
.\.venv\Scripts\python.exe -B tools\story_archive_manager.py `
  --run-dir outputs\runs\job_001 `
  --archive-db outputs\archive\archive.sqlite3
```

### 归档成功后删除源运行库

```powershell
$env:PYTHONDONTWRITEBYTECODE='1'
.\.venv\Scripts\python.exe -B tools\story_archive_manager.py `
  --run-dir outputs\runs\job_001 `
  --archive-db outputs\archive\archive.sqlite3 `
  --delete-source-db
```

### 通过统一 CLI 调用

```json
{
  "action": "archive_run",
  "payload": {
    "run_dir": "outputs/runs/job_001",
    "archive_db_path": "outputs/archive/archive.sqlite3",
    "delete_source_db": true
  }
}
```

## 归档校验

归档器写入后会做最基本的自校验，至少确认下面这些点：

- `archive_jobs` 中能查到当前 `job_id`
- 各业务归档表的记录数与源运行库一致
- 最终选中的 pack / plan / payload / draft 都能查到
- 选中的 draft 有非空 `content_markdown`

只有这轮校验通过后，才适合删除源运行库。

## 当前限制

目前归档器还有两个明确边界：

### 1. 只支持单 job 单 case 的 report

如果一个 `report.json` 里同时塞了多个 case，当前版本会直接拒绝归档。

### 2. 归档库不应参与生成

`archive.sqlite3` 只负责沉淀，不应该被拿来继续跑：

- `build_idea_packs`
- `build_story_plans`
- `build_story_drafts`

生成阶段仍然应该使用独立运行库。

## 推荐流程

### 并发生成

1. 从模板库初始化任务运行库
2. 用运行库完成整条写作链
3. 生成 `report.json`

### 串行归档

1. 归档器把运行库写入 `archive.sqlite3`
2. 校验归档结果
3. 校验通过后删除运行库

这样可以同时满足：

- 运行阶段强隔离
- 归档阶段统一检索
- 目录数量可控
