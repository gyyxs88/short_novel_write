# 工作流

本文档用于约束 `short_novel_write` 的执行顺序。  
目标不是把所有创意都写死，而是保证短篇写作有一条稳定、可复用的主流程。

## 总原则

- 先收敛创意，再展开正文
- 用户不要求逐步确认时，默认自动继续
- 先保证故事完整，再追求文风包装
- 第一阶段优先本地成稿，不默认绑定平台

## 入口类型

触发任务后，先判断当前请求属于哪一种：

1. 从零开始写一篇短篇
2. 只有脑洞，需要补创意
3. 已有创意，需要补方案
4. 已有方案，需要写正文
5. 已有正文草稿，需要续写或修订

不同入口都尽量汇入同一条主流程，不要每种情况各写一套完全不同的逻辑。

## 标准主流程

### 第 1 步：收集并整理输入

需要尽快确认以下信息里哪些已经给出：

- 题材或方向
- 关键词或情绪
- 主角设定
- 核心关系
- 目标篇幅
- 是否希望自己参与选择

如果用户已经给了明确信息，不重复追问同一项。

### 第 2 步：创意阶段

如果用户没有可直接落地的核心创意：

- 生成 `4` 组候选创意
- 候选之间至少在下面两项上明显不同：
  - 题材方向
  - 核心关系
  - 主冲突
  - 转折方式

每组创意最少输出：

- 一句话钩子
- 一句话故事内核
- 主冲突
- 关系亮点
- 推荐题材标签

如果用户没有明确说要自己选，就自动保留最稳妥的一组继续。

### 第 3 步：篇幅阶段

默认给出固定篇幅档位：

- `3000-5000` 字，约 `4-6` 章
- `5000-8000` 字，约 `6-8` 章
- `8000-12000` 字，约 `8-10` 章
- `12000-15000` 字，约 `10-12` 章

默认值：

- `5000-8000` 字
- `6-8` 章

如果用户只说“写一篇短篇”，且没有额外限制，就直接使用默认值继续。

### 第 4 步：方案阶段

围绕已确定的创意，生成 `3-4` 组故事方案。  
每组方案必须包含：

- 暂定标题
- 一句话卖点
- 主角目标
- 核心关系
- 主冲突
- 关键转折
- 结尾落点
- 章节节奏概览

标题要求：

- 不直接复读用户原句
- 优先 `6-16` 个汉字
- 要具体，不要太空
- 有画面感或冲突感

如果用户不选，自动保留最有吸引力且最容易写成完整短篇的一组。

如果仓库内已有 `tools/story_cli.py` 且当前创意包已经收敛：

1. 优先调用 `build_story_plans`
2. 再用 `list_story_plans` 读取候选方案
3. 如果用户不选，默认自动选一组最适合短篇节奏的方案
4. 选定后可用 `update_story_plan_status` 记录当前状态
5. 如果走 LLM 链，不要默认和正文阶段共用同一个 `llm_environment`
6. 对豆瓣风格或其他长输出更敏感的风格，优先给方案阶段单独准备一个偏稳妥的环境

### 第 5 步：写作简报阶段

开始正文前，先把当前方案压缩成一个最小写作简报。  
最少应包含：

- 标题
- 目标字数
- 目标章节数
- 主角
- 核心关系
- 主冲突
- 转折点
- 结尾方向

这一步不要求用户单独确认，但内部必须先收敛，否则正文容易越写越散。

如果仓库内已有 `tools/story_cli.py`：

1. 优先调用 `build_story_payloads`
2. 再用 `list_story_payloads` 读取正文 payload
3. 后续正文阶段默认直接消费 payload，不要再从方案全文临时抽字段

### 第 6 步：正文阶段

默认先写简介，再写正文。

默认本地成稿结构：

```markdown
# 标题

## 简介

50-120 字的钩子式简介

## 正文

### 1
### 2
### 3
```

正文阶段要求：

- 开头前几段要尽快进入冲突或异常
- 每章都要推进事件、关系或认知
- 中段至少要有一次有效偏转
- 章尾尽量保留继续读下去的理由
- 结尾要回收核心冲突，不写成断头文

如果仓库内已有 `tools/story_cli.py`：

1. 优先调用 `build_story_drafts`
2. 再用 `list_story_drafts` 读取 Markdown 草稿
3. 默认在已有草稿上继续修订，不要重新起一版自由文本
4. 如果需要标记当前采用版本，可用 `update_story_draft_status`
5. 如果走 LLM 链，正文阶段可以和方案阶段使用不同 `llm_environment`
6. 对豆瓣风格，正文环境默认要比方案环境更保守，优先给更长 timeout，不要直接复用知乎或方案阶段的环境
7. 如果当前目标是直接生成可进入自检和保存的版本，优先在 `build_story_drafts` 时开启：
   - `auto_revise=true`
   - 知乎风传 `revision_profile_name="zhihu_tight_hook"`
   - 豆瓣风传 `revision_profile_name="douban_subtle_scene"`
8. 只有在需要保留原始首稿做对比时，才关闭 `auto_revise`

### 第 7 步：自检与修订阶段

正文完成后，按 `quality-checklist.md` 自检。  
至少处理下面几类问题：

- 字数不对
- 结构不完整
- 开头太慢
- 中段拖沓
- 转折生硬
- 结尾无回收
- 标题不贴题

如果仓库中存在 `tools/story_cli.py`，这里优先走 CLI：

1. 用 `inspect` 动作传入当前成稿内容
2. 读取返回 JSON 里的 `overall_ok`
3. 分别检查 `structure.issues` 和 `quality.issues`
4. 结合 `quality.suggestions` 做小步修订

如果发现问题，优先做小步修订，不要推翻重写整篇。

如果当前正文来自 `build_story_drafts`：

- 先修当前草稿
- 再重复执行 `inspect`
- 不因为一轮检查失败就放弃既有稿件

如果当前正文来自 `build_story_drafts` 且已开启 `auto_revise`：

- `inspect` 默认直接检查修订后的 draft 主记录
- 先看自动修订后版本还有哪些结构或质量问题
- 只在自动修订仍未过关时，才额外手动调用 `revise_story_draft`

### 第 8 步：保存阶段

默认落盘到：

- `outputs/novels/`

保存要求：

- `UTF-8`
- `.md`
- 文件名使用最终标题

如果已经有工具：

- 优先通过 `tools/story_cli.py` 的 `save` 动作保存
- 保存后记录 `output_path`

如果还没有工具：

- 直接保存，不阻塞任务

## 工具接入点

下面这些环节适合工具接入：

- 创意去重或相似度提示
- 方案生成
- 正文 payload 组装
- 正文草稿生成
- 标题候选检查
- 字数统计
- 章节结构检查
- 文件写入

下面这些环节仍应主要由技能控制：

- 创意判断
- 方案取舍
- 写作推进
- 文风与节奏判断

## 当前边界

本工作流默认只覆盖“本地短篇写作”这条主线。  
如果后续要加平台发布能力，应作为独立扩展，而不是直接把这里改成平台操作说明书。
